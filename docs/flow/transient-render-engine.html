Transient Render Engine

This article is an introduction to the â€‹TRE architecture.

Element Models#

Element models form the building block of the engine. These models specify how a DOM element of a peculiar tag should be translated. You can tamper with those models and add you own models, making this library extremely customizable.

HTMLElementModel#

To each standard tag is attached an element model, instance of the â€‹HTMLElementModel class. Such model has multiple fields describing different behaviors related to translation of those DOM elements:

â€‹contentModel

How should this tag be translated? See next chapter.

â€‹isVoid

Will be true for void elements , e.g. DOM elements which can't have children.

â€‹isOpaque

Will be true for those elements which children should not be translated. Useful for â€‹<svg> and other custom markups.

â€‹mixedUAStyles

Mixed User-Agent styles, e.g. default styles for this element. This is how default styles are set for tags.

â€‹getUADerivedStyleFromAttributes

A function which returns mixed UA styles given the DOM node attributes and â€‹TNode â€‹Markers.

HTMLContentModel#

There are 4 content models that can be attached to a tag:

â€‹HTMLContentModel.textual

For elements which can be translated to â€‹TText or â€‹TPhrasing. Examples: â€‹<span>, â€‹<strong> ...

â€‹HTMLContentModel.block

For elements which can only be translated to â€‹TBlock. Examples: â€‹<div>, â€‹<p>, â€‹<article> ...

â€‹HTMLContentModel.mixed

(rare) for elements which can be translated to â€‹TText, â€‹TPhrasing or â€‹TBlock. The sole mixed elements are â€‹<a>, â€‹<ins> and â€‹<del>.

â€‹HTMLContentModel.none

For element which shall not be rendered and will be translated to â€‹TEmpty. Examples: â€‹<button>, â€‹<map> ...

A powerful feature of the Foundry engine is that the models attached to a tag name can be customized! See the â€‹Custom Rendering page.

Steps#

The â€‹TRT construction is broadly comprised of three steps.

Translation#

Each DOM element is translated to a â€‹TNode. The translation will obide by the following rules:

  1. The root of the document will be translated to a â€‹TDocument node. This node has a special context field which holds metadata harvested in the â€‹<head> DOM element (see â€‹DocumentMetadata).

  2. Text nodes will be translated to â€‹TText, and will be merged with a parent DOM element if the parent's content model is textual or mixed when they are its only child. For example, a Text node with no siblings which parent is a â€‹<span> will be merged into a â€‹TText with tagName set to "span".

  3. DOM elements which content model is textual with multiple children will be translated to â€‹TPhrasing nodes.

  4. DOM elements with children which content model is mixed will be translated to â€‹TPhrasing if they only have â€‹TPhrasing or â€‹TText children, â€‹TBlock otherwise.

  5. DOM elements which content model is block will be translated to â€‹TBlock nodes.

  6. Finally, DOM elements which content model is none will be translated to â€‹TEmpty.

  7. â€‹<script>, comments and â€‹<style> DOM nodes will be ignored.

note

A DOM element might be untranslatable for a variety of reasons. For example, its name is not a standard HTML5 element and there is no custom HTML element model registered for it. An other reason is that it is an interactive element such as a form, input or button.

In addition, inline styles, User Agent styles and mixed styles are processed by the CSS Processor, see â€‹CSS Processing for more details.

Hoisting#

The hoisting phase consists in enforcing a basic constraint:

The Hoisting Constraint

A â€‹TPhrasing node should only have â€‹TText, â€‹TPhrasing and â€‹TEmpty nodes as children.

Therefore, any â€‹TBlock child of a â€‹TPhrasing node will be recursively hoisted to the parent, until it meets that constraint. This constraint must be enforced to insure that a React Native â€‹Text elements have no â€‹View children, since it will break the default box model and might lead to bugs and inconsistencies. The View Constraint

React Native â€‹Text elements should not have â€‹View elements as children.

On one hand â€‹TBlock will be translated to â€‹View elements and on the other hand â€‹TPhrasing and â€‹TText nodes will be translated to â€‹Text elements. Therefore, enforcing The Hoisting Constraint in the â€‹TRT results in enforcing The View Constraint at render time. You can disable hoisting via â€‹dangerouslyDisableHoisting prop, but be advised this is yet experimental.

Anatomy of a TNode#

A â€‹TNode has the following relevant fields (see â€‹TNodeShape for a reference):

â€‹attributes

The list of attributes attached to the underlying DOM Node.

â€‹id

The id attached to the underlying DOM Node.

â€‹classes

An array of classes associated with the underlying DOM Node.

â€‹domNode

The underlying DOM Node, if present.

â€‹tagName

The tag name attached to the underlying DOM Node.

â€‹parent

The parent â€‹TNode, if present, determined before hoisting.

â€‹nodeIndex

The position of this element relative to its parent, before hoisting and after whitespace collapsing.

â€‹children

An array of â€‹TNode descendents to this node.

â€‹type

The type of this â€‹TNode. Either text, phrasing, block, document or empty.

â€‹markers

A registry of markers for this â€‹TNode. See explaination in below section.

â€‹hasClass

A utility function to check if this node has the provided CSS class.

â€‹snapshot

A utility function to create a JSX-like string representation of this node and its children. Very handy for debugging.

warning

The styles field which is not listed here is not consumable as a React Native component style prop.

Markers#

â€‹Markers form an abstraction in which one â€‹TNode provides semantic information to itself and all its descendants. For example, â€‹<ins> elements, which stand for "insertion" of content in the context of an edit will provide the edits marker with value "ins" to all its descendants. Similarly, â€‹<a>, â€‹<ol> and â€‹<ul> elements will set their own markers. List markers such as olNestLevel are integers which are incremented each time a list is nested.

Markers can also be derived from attributes. This is the case with â€‹dir and â€‹lang attributes. Finally, you can customize the markers extraction logic with â€‹setMarkersForTNode prop.
